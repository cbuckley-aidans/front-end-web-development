Creating User Stories from Technical Requirements

User stories help translate technical requirements into clear, user-focused descriptions of functionality. They follow the format:

As a [type of user], I want to [goal], so that I can [benefit].

Steps to Creating User Stories

Identify the User(s)
Define Goals
Specify Benefits

1. Identify Users

Who will use this feature? Be specific about the type of user:

  • Registered user
  • Admin user
  • Guest visitor

2. Define Goals

What does the user want to accomplish? Make it:

  • Clear and specific
  • Action-oriented
  • Achievable

3. Specify Benefits

Why does the user want this? Include:

  • Clear value
  • Business benefit
  • User benefit

Example 1: Keeping Stories Focused

‚ùå Bad User Story:

As a student, I want to log in to the school website, view my timetable, check my grades, and message my teachers so that I can manage my school work.

Why this needs improvement:

This story tries to do too many things at once. It combines login, timetable, grades, and messaging into one story.

‚úÖ Better User Story:

As a student, I want to view my current timetable so that I can see my daily class schedule.

Why this is better:

This story focuses on one specific feature (viewing timetable). Other features like login and messaging would be separate stories.

Example 2: Writing for Users

Show Bad Example Show Good Example
‚ùå Bad User Story:

The system needs to implement a MongoDB database to store user profile pictures with image compression.

Why this needs improvement:

This is written as a technical requirement, not a user story. It focuses on the technical implementation (MongoDB, compression) rather than user needs.

‚úÖ Better User Story:

As a student, I want to upload a profile picture to my account so that my classmates can easily recognize me.

Why this is better:

This story focuses on what the user wants to do (upload a picture) and why they want to do it (recognition by classmates).

Example 3: Being Specific

Show Bad Example Show Good Example
‚ùå Bad User Story:

As a student, I want the website to work better so that I can use it more easily.

Why this needs improvement:

"Work better" and "more easily" are too vague. How would we know when this story is complete?

‚úÖ Better User Story:

As a student, I want to search for assignments by subject and due date so that I can prioritize my homework.

Why this is better:

This story clearly states what the user wants (search by subject/date) and why (prioritize homework). We can test if these specific features work.

Creating Design Criteria

Once you've gathered your user stories, you can transform them into clear design criteria that will guide your website development and evaluation.

Remember:

Design criteria should focus on WHAT the website must do, not HOW it will do it.

Example: From User Story to Design Criteria

üìù User Stories

  • "As a student, I want to search for assignments by subject and due date so that I can prioritize my homework."
  • "As a student, I want to see my assignment grades immediately after they're entered so I can track my progress."

‚úÖ Design Criteria

  • The website must allow users to filter assignments by both subject and due date
  • The website must display assignment grades to students as soon as they are entered into the system
